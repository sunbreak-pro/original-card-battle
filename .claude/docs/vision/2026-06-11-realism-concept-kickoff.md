## 4. このセッションで決めること（テーマ決め = 決定事項リスト）

### 4.1 最優先: 「リアル性」の定義（複数選択可・優先順位付き）

ユーザーに提示する解釈候補。**どの軸を・どの強度で**入れるかを合意する:

1. **身体のリアリティ** — 負傷部位 / 出血 / 疲労 / 空腹 / 治療に時間がかかる。HPという抽象値からの脱却
2. **戦闘のリアリティ** — 間合い・体勢・武器の重さ・防具の部位。カード=「実際に取れる行動」の表現
3. **世界・生活のリアリティ** — 経済（物価・需給）/ 時間経過 / 天候 / NPC の生活。ダンジョン外の世界が生きている
4. **結果の重さ（permadeath 系）** — 死・喪失が重い。回復が安くない。選択の不可逆性
5. **情報のリアリティ** — 敵のステータスが最初は見えない / 観察・経験で判明する。数値の非開示
6. **アナログ TCG 的リアリティ** — カードの physicality（質感・希少性・トレード感）。演出面（PixiJS 活用先）
7. **確率の透明性 / 物理挙動** — ダイス・物理演算の見える化（リアル「っぽさ」の演出）

→ 成果: **「本作におけるリアル性とは \_\_\_ である」の1〜3文 + 採用軸の優先順位表**

### 4.2 コアファンタジーと体験の柱

- プレイヤーは何者で、何をする体験か（1行）
- 体験の柱 3本（例: 「読み合い」「喪失の緊張」「成長の手応え」— リアル性軸と整合させる）
- トーン・世界観（現状はダークファンタジー寄り: 腐食/狂乱/混沌/虚無/深淵。維持か変更か）

### 4.3 既存資産との関係（A/B/C 分岐）

- **A 増築**: 既存ループ維持 + リアル性システムを足す（工数小・一貫性リスク）
- **B 再設計**: コアループから見直し、データ・UI・PixiJS 基盤を流用（工数大・一貫性高）
- **C 並立**: 新モード or 新作品として切り出し（既存を壊さない・保守2本化）

判断材料: 採用したリアル性軸が既存ループ（ラン制ローグライク + キャンプ）と矛盾するか否か。

### 4.4 スコープ境界

- バックログ既存機能（AoE / Quest / Title / NPC会話 / Dark Market / PixiJS Phase 2-4）の扱い: 新要件に統合 / 凍結 / 廃棄
- 技術的負債（known-issue 001 / rollup ハードコード / テスト負債）をどの段階で返済するか
- 「ほぼ完璧な実装」の合格基準（例: 設計書と実装の差分ゼロ / テストカバレッジ基準 / 通しプレイで破綻なし）

## 5. セッションの進行手順（推奨）

1. **インタビュー**: §4.1 → 4.2 → 4.3 → 4.4 の順で AskUserQuestion を使い1テーマずつ確定（一括質問しない。前の回答が次の選択肢を変えるため）
2. **コンセプトシート作成**: 決定事項を `docs/vision/core.md` の改訂版（または新規 `docs/vision/concept-v2.md`）としてまとめ、ユーザー承認を取る
3. **ギャップ分析**: 承認済みコンセプトと現実装/設計書群（`docs/*_document/`）の差分を洗い出し、影響範囲表を作る
4. **要件定義書**: `docs/requirements/` に Tier 1（コア）/ Tier 2（補助）/ Tier 3（実験）で記述
5. **実装プラン**: `docs/vision/plans/YYYY-MM-DD-<slug>.md` に分割（各ステップ1セッション規模・Gate 列付き）。MEMORY.md へ task-tracker で起票

注意:

- 手順 1-2 が本丸。**コンセプト合意前に実装やギャップ分析へ進まない**
- 数値・仕様の正は設計書（設計書駆動）。要件が変わったら設計書を先に直す
- Non-Goals を覆す提案（マルチプレイ化等）はしない。覆したい場合はユーザーから言い出すのを待つ
